Arhitectură · Workflow automatizat

Cum funcționează pipeline-ul HR de la inbox la dashboard, end-to-end

Aplicația automatizează patru pași HR care în mod normal consumă cel mai mult timp: procesarea cererilor de concediu, sincronizarea calendarului partajat, calculul scorurilor de retenție pe persoană și verificarea expirării certificatelor. Fiecare strat e o componentă Python independentă, conectate de un singur orchestrator. Dashboard-ul HTML și un server MCP expun rezultatul către utilizatorii umani și, respectiv, către orice LLM conectat.

Etapele automatizării

EMAIL · GMAIL / OUTLOOK

Ingestia cererilor de concediu

Un job rulează la 5 minute, citește inboxul prin API, identifică emailurile cu cereri de concediu (medical, vacanță, training, personal). Pentru fiecare email matched:

  • extrage tipul prin regex pe subiect + corp (cuvinte-cheie RO + EN)
  • extrage perioada (de la → până la) din text + atașament OCR-ready
  • matched expeditorul la un angajat din director
  • scrie un rând în data/time_off.csv + audit log JSONL

Idempotent — cheia primară e SHA-1 pe message-id, re-rulările nu dublează niciodată un email.

Sincronizarea calendarului partajat

Sincronizează fereastra de 14 zile din calendarul HR partajat (training, all-hands, review cycles, concedii înregistrate). Pentru fiecare zi:

  • numără evenimentele și clasifică categoria dominantă
  • calculează intensitatea (0–3) pentru heatmap-ul din sidebar
  • extinde evenimentele multi-day (ex: concediu de 5 zile) în ocurențe per zi

Provider pattern: MockGraphProvider pentru demo, GraphCalendarProvider documentat pentru OAuth + endpoint real.

scripts/sync_calendar.pydata/calendar.json
AI · MODEL DE ANOMALIE

Scor de risc de plecare

IsolationForest nesupervizat antrenat pe 6 feature-uri inginerie: vechime, raport salariu vs piață, scor de performanță, zile de la ultimul review, număr de promovări, ore de training YTD.

  • scor normalizat în [0, 1] per angajat
  • etichetă low / medium / high derivată din praguri
  • top trei semnale (z-score) pentru încadrarea conversației 1:1

Determinist via random_state=42 și contamination=0.2.

Sănătatea echipei

KMeans cu k=3 peste 5 metrici de echipă: scor mediu de performanță, realizare KPI, procent de proiecte la timp, completare medie a proiectelor, raport mediu de salariu.

  • etichete: Thriving · Steady · At-risk
  • fiecare cluster mapat la un playbook concret de acțiuni
  • numirea clusterelor se face pe centroid (nu hard-coded)

Ce face workflow-ul să fie production-ready

Determinism

Aceleași input-uri produc același output — byte-identic. Fiecare model sklearn folosește random_state=42, fiecare JSON e scris cu sort_keys=True.

Idempotență

Job-ul de email se poate rula de oricâte ori — duplicatele sunt detectate prin hash pe message-id și sărite. Sigur pentru re-rulări automate.

Pista de audit

Fiecare email procesat lasă o linie JSON în data/time_off_audit.jsonl cu timestamp + payload complet — util pentru audituri legale.

Protocol-first

Integrările (Gmail, Microsoft Graph) sunt definite ca Protocol Python — mock-uri pentru demo, swap într-o linie pentru producție. Zero coupling cu un provider anume.

Layer LLM · Server MCP

Model Context Protocol — expunere către LLM-uri

Stub-ul din scripts/mcp_server.py publică datele dashboard-ului ca 6 resurse + 4 tool-uri conform standardului MCP. Un client compatibil (Claude Desktop, agent runtime intern, etc.) se conectează și poate interoga direct cu schema corectă — fără să re-implementeze stratul de date.

RESURSE EXPUSE
hr://employees hr://attrition hr://teams hr://compliance hr://calendar hr://inbox
TOOL-URI APELABILE
hr.who_is_at_risk(threshold, team_id?) hr.team_health(team_id) hr.compliance_gaps(within_days, cert_type?) hr.upcoming_calendar(days, category?)

Astfel, întrebarea „Cine ar trebui programat la 1:1 săptămâna asta?" devine o invocare structurată hr.who_is_at_risk(threshold=0.65) — răspuns garantat în schema documentată, nu prompt-engineering peste un CSV brut.

Architecture · Automated workflow

How the HR pipeline runs end-to-end, from inbox to dashboard

The application automates four HR steps that normally consume the most time: processing leave requests, syncing the shared calendar, computing per-person retention risk scores, and checking certification expiry. Each layer is an independent Python component, glued together by a single orchestrator. The HTML dashboard and an MCP server expose the result to human users and any connected LLM, respectively.

Automation stages

EMAIL · GMAIL / OUTLOOK

Leave-request inbox ingestion

A job runs every 5 minutes, reads the inbox via API, and identifies leave-request emails (medical, vacation, training, personal). For each matched email it:

  • extracts the type via regex on subject + body (RO + EN keywords)
  • extracts the date range from the body text + OCR-ready attachment
  • matches the sender to an employee in the directory
  • writes a row to data/time_off.csv + JSONL audit log

Idempotent — primary key is a SHA-1 of the message-id; re-runs never duplicate an email.

scripts/ingest_emails.pydata/time_off.csvdata/time_off_audit.jsonl
CALENDAR · MICROSOFT GRAPH

Shared calendar sync

Syncs a 14-day window from the shared HR calendar (training, all-hands, review cycles, recorded leave). For each day it:

  • counts events and classifies the dominant category
  • computes intensity (0–3) for the sidebar heatmap
  • expands multi-day events (e.g. 5-day leave) into per-day occurrences

Provider pattern: MockGraphProvider for the demo, GraphCalendarProvider documented for OAuth + real endpoint.

scripts/sync_calendar.pydata/calendar.json
AI · ANOMALY MODEL

Retention-risk scoring

Unsupervised IsolationForest trained on 6 engineered features: tenure, comp ratio vs market, performance score, days since last review, promotion count, training hours YTD.

  • score normalized in [0, 1] per employee
  • label low / medium / high derived from thresholds
  • top three signals (z-score) for framing the 1:1 conversation

Deterministic via random_state=42 and contamination=0.2.

scripts/score_attrition_risk.pydata/ai_attrition.json
AI · CLUSTERING

Team health

KMeans with k=3 over 5 team-level metrics: average performance score, KPI attainment, on-track project rate, average completion, average comp ratio.

  • labels: Thriving · Steady · At-risk
  • each cluster maps to a concrete recommended-action playbook
  • cluster naming derives from centroid magnitude (not hard-coded)
scripts/cluster_team_health.pydata/ai_recommendations.json
COMPLIANCE · LABOR LAW

Certification check

Reads expiry dates for four mandatory certification types and flags anything expiring within the warning window:

  • periodic medical check · occupational safety (SSM) · first aid · management training (for leads)
  • separate flag for hours over the legal cap (1800h YTD)
  • output ordered by urgency: expired > expiring < 60 days > valid

What makes the workflow production-ready

Determinism

The same inputs produce the same output — byte-identical. Every sklearn model uses random_state=42, every JSON is written with sort_keys=True.

Idempotency

The email job can run any number of times — duplicates are detected by message-id hash and skipped. Safe for automated re-runs.

Audit trail

Every processed email appends a JSON line to data/time_off_audit.jsonl with timestamp + full payload — useful for legal audits.

Protocol-first

Integrations (Gmail, Microsoft Graph) are defined as Python Protocol abstractions — mocks for the demo, one-line swap for production. Zero coupling to any specific provider.

LLM layer · MCP server

Model Context Protocol — exposing data to LLMs

The stub at scripts/mcp_server.py publishes the dashboard's data as 6 resources + 4 tools following the MCP standard. A compatible client (Claude Desktop, internal agent runtime, etc.) connects and can query directly with the right schema — without re-implementing the data layer.

EXPOSED RESOURCES
hr://employees hr://attrition hr://teams hr://compliance hr://calendar hr://inbox
CALLABLE TOOLS
hr.who_is_at_risk(threshold, team_id?) hr.team_health(team_id) hr.compliance_gaps(within_days, cert_type?) hr.upcoming_calendar(days, category?)

That way, the question "Who should I schedule a 1:1 with this week?" becomes a structured call to hr.who_is_at_risk(threshold=0.65) — guaranteed answer in the documented schema, not prompt-engineering over a raw CSV.
